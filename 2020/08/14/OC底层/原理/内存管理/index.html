    CADisplayLink、NSTimer 定时器

CADisplayLink、NSTimer 会对 target 产生强引用，如果 target 又对它们产生强引用，那么就会引发循环引用。

使用注意

CADisplayLink

CADisplayLink 保证调用 -(void)timerTest 方法的频率和屏幕的刷新帧频率一致，60FPS。实际的调用频率可能不太一样，根据任务的耗时情况会有减少。

解决方案

使用 block

打印结果：

在使用 NSTimer 的时候，使用 block 的方式传入 weakSelf 可以有效解决循环引用的问题。并且以 scheduled 开头的方法会自动将 timer 添加到当前的 runloop 中并使用 default mode。

target 传入 weakSelf 并不能解决循环引用个问题，因为 NSTimer 内部对传入的 target 也是强引用的，而且 weakSelf 只是用来解决 block 循环引用问题的方案。

CADisplayLink 没有 block 相关的 API。

使用中间对象

定义中间对象：

给定时器的 target 传入 YQProxy 对象，在 YQProxy 对象尝试调用 -(void)timerTest 方法时，发现没有实现会走消息转发的逻辑，会调用到 - (id)forwardingTargetForSelector:(SEL)aSelector 方法，在该方法内部返回实现了 -(void)timerTest 方法的对象，就可以正常实现 timer 定时器调用 -(void)timerTest 方法的逻辑了。

使用代理对象（NSProxy）

定义 YQTimerProxy 继承自 NSProxy：

给定时器的 target 传入 YQTimerProxy 对象，YQTimerProxy 对象不会尝试调用 -(void)timerTest 方法，而是直接走消息转发逻辑，返回已经实现了 -(void)timerTest 方法的对象。

如果 YQTimerProxy 中没有添加消息转发的逻辑会出现如下报错：

可以看到 YQTimerProxy 没有去查找 -(void)timerTest 方法，而是直接查找的 - (NSMethodSignature *)methodSignatureForSelector:(SEL)sel 方法。这一点可以通过 GNUStep 源码找到原因，NSProxy 内部的方法实现都是直接调用的 - (NSMethodSignature *)methodSignatureForSelector:(SEL)sel 方法。
